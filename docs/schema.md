# Database Schema

## Overview

MarketMint uses a **SQL relational database** (PostgreSQL-compatible) as its persistence layer. Firestore is not used.

Authentication is handled by **Better Auth**, an internal backend library. Better Auth auto-generates and manages its own auth tables (`user`, `session`, `account`, `verification`). MarketMint extends these with application-level fields and owns the domain tables.

The schema is represented in [`schema.dbml`](./schema.dbml) (visualize at [dbdiagram.io](https://dbdiagram.io)).

See [`docs/auth-security.md`](../auth-security.md) for the full authentication design.

---

## Key Architectural Decisions

### SQL, not Firestore

The persistence layer is a SQL relational database. The schema uses standard relational modeling: primary keys, foreign keys, unique constraints, and indexes.

### Better Auth as the authentication foundation

Better Auth runs inside the MarketMint backend and stores auth data in the same SQL database. It is not an external hosted provider. It manages:
- User identity (`user` table)
- Cookie-based sessions (`session` table)
- Provider accounts â€” email/password and Google OAuth (`account` table)
- Email verification (`verification` table)

MarketMint does not duplicate these with custom `auth_identities`, `password_credentials`, or `user_sessions` tables.

### MarketMint extends the Better Auth user

Better Auth supports additional fields on the `user` table. MarketMint adds:
- `role` â€” `user` or `admin`
- `banned` / `banReason` / `banExpires` â€” managed by Better Auth's Admin plugin

These are additional columns on the Better Auth `user` table, defined in `apps/api/src/db/schema.ts`.

### Cookie-based sessions

Better Auth uses secure HTTP-only cookies for browser sessions. Tokens are not stored in `localStorage`. Cookies work correctly in incognito/private browsing and are cleared when the private session ends.

### Local synchronized crypto catalog

The backend fetches up to 250 assets from CoinGecko every 10 minutes and stores them in `crypto_assets`. The frontend never calls CoinGecko directly. All dashboard data, search, favorites, and alerts operate against this local catalog.

### MVP currency: USD only

All monetary values are in USD. Multi-currency support is a future improvement.

---

## Design Principles

- **Delegate standard auth to Better Auth.** Password hashing, session management, and OAuth provider handling are Better Auth's responsibility. MarketMint does not reinvent these.
- **Extend, don't duplicate.** MarketMint adds application-level fields (`role`, `banned`, `banReason`, `banExpires`) to the Better Auth user. It does not create a competing user table.
- **All user references use the Better Auth user id.** `user_profiles`, `user_favorite_cryptos`, `audit_logs`, and `price_alerts` all reference `user.id` from Better Auth as the canonical user identifier.
- **Crypto market data is shared, backend-owned data.** Prices and KPIs are populated by the sync process on a schedule, not on user request.
- **Keep admin-editable fields limited.** Admins can update `name`, `role`, and ban state. They cannot touch credentials, provider accounts, or `created_at`.
- **USD is fixed for the MVP.**

## Access Rules

| Actor | Permission |
|---|---|
| Authenticated user | Read/write own `user_profiles`, `user_favorite_cryptos`, `price_alerts` |
| Authenticated user | Read `crypto_assets` and `crypto_market_kpis` via backend API |
| Authenticated user | Cannot read other users' data |
| Admin | List and read all users (via Better Auth admin or custom endpoint) |
| Admin | Update `name`, `role`, `status` on any user |
| Admin | Block, unblock, deactivate, delete users |
| Admin | Cannot modify Better Auth-owned credential or provider fields |
| Backend only | Write `crypto_assets`, `crypto_market_kpis`, `crypto_sync_runs`, `audit_logs` |
| Backend only | Communicate with CoinGecko â€” never the frontend |

## Crypto Data Synchronization

### Sync flow

```
Scheduled job or internal trigger
â†’ POST /api/internal/crypto/sync  (protected by CRYPTO_SYNC_SECRET)
â†’ Backend calls CoinGecko GET /coins/markets (per_page=250, vs_currency=usd)
â†’ Backend calls CoinGecko GET /global
â†’ Backend normalizes responses
â†’ Backend upserts crypto_assets (one row per coin, up to 250)
â†’ Backend marks assets absent from response as is_active = false
â†’ Backend upserts crypto_market_kpis
â†’ Backend inserts crypto_sync_runs record
â†’ Dashboard reads from the database
```

One complete sync uses **2 CoinGecko API calls**.

### API usage estimate

| Metric | Value |
|---|---|
| Calls per sync | 2 |
| Syncs per hour | 6 |
| Calls per day | 288 |
| Calls per 30-day month | 8,640 |

Stays within a 10,000 monthly call budget.

### Dashboard read flow

```
GET /api/crypto/dashboard
â†’ requireRole() middleware validates Better Auth session and rejects banned users
â†’ Backend reads crypto_market_kpis
â†’ Backend reads crypto_assets WHERE is_active = true ORDER BY market_cap_rank ASC LIMIT 10
â†’ Backend returns { kpis, assets, lastUpdatedAt, isStale }
```

**Stale threshold:** `is_stale = true` if `last_synced_at` is older than 15 minutes.

### Local search flow

```
GET /api/crypto/assets/search?q=...
â†’ requireRole()
â†’ Backend queries crypto_assets WHERE is_active = true AND (name ILIKE or symbol ILIKE)
â†’ No CoinGecko call
```

### Favorites flows

```
POST /api/me/favorite-cryptos/:providerAssetId
â†’ requireRole()
â†’ Verify crypto_assets.provider_asset_id exists and is_active = true
â†’ Insert user_favorite_cryptos row

GET /api/me/favorite-cryptos
â†’ requireRole()
â†’ Read user_favorite_cryptos for current user
â†’ JOIN crypto_assets for current market data
â†’ Return favorites with live local market values
```
